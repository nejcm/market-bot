# Surgical ADR Improvement Program — Implementation Handoff

## Purpose

Implement the confirmed correctness and safety fixes without introducing speculative artifact
schemas, lifecycle abstractions, calibration statistics, or performance infrastructure.

The repository is a single-user, US-centric research tool with 15 historical run directories,
26 persisted forecasts, and no resolved forecasts. Keep each change independently reviewable and
consistent with the repository's scope-discipline rule.

## Repository constraints

- Research-only: no buy/sell/hold, sizing, or execution language.
- Predictions remain observable from public data.
- No secrets in code, tests, fixtures, traces, or errors.
- Bun and oxc only.
- No bundled refactors or speculative abstractions.
- Preserve unrelated working-tree and `plans/` changes.
- Add tests with each behavioral change.
- Run `bun run check` at each stable completion point and before completion.

## Implementation sequence

Land each numbered stage as its own reviewed commit with tests.

### 1. Universal external-text safety

- Reuse or generalize the existing model-visible web-text sanitizer only as required for reuse; do
  not perform a terminology-only rename.
- Sanitize all provider-normalized prose before it enters model context:
  - titles
  - publishers
  - summaries and descriptions
  - snippets
  - SEC filing excerpts
- Cover MarketAux, Finnhub, Yahoo, Massive, SEC, and Exa normalization paths.
- Validate rather than prose-sanitize identifiers, dates, and URLs.
- Preserve raw provider snapshots byte-for-byte.
- Preserve safe retained text and existing sanitizer telemetry conventions.
- When no safe prose remains, omit it from model-visible evidence and emit a Source Gap without
  discarding the raw snapshot.

Tests:

- Add a provider fixture for every named normalization path.
- Verify instruction spans and unsafe HTML do not reach model-visible values.
- Verify safe remainder and sanitizer counts.
- Verify empty sanitized content creates a gap.
- Verify raw snapshots remain unchanged.

### 2. Frozen Source Plan before collection

Keep the current v2 artifact family:

- `normalized/source-plan.json`
- `normalized/evidence-lanes.json`
- `normalized/source-ledger.json`

Split the current post-hoc builder into two responsibilities:

- `buildSourcePlan(command, generatedAt)` creates the immutable v2 Source Plan before
  `collectSources`.
- `assessSourcePlan(plan, collectedSources)` creates Evidence Lanes and the Source Ledger after
  collection.

Requirements:

- Resolve the research command and checked-in research subject before creating the plan.
- Capture `generatedAt` and build the plan before the first source-provider I/O.
- Derive lane applicability and evidence class only from the resolved command, subject, asset
  class, depth, and checked-in policy.
- Never derive planned applicability or evidence class from collected outcomes, credentials,
  provider availability, or successful fetches.
- Use existing lane names as stable requirement identities.
- Grade coverage, gaps, freshness, and corroboration against the frozen plan.
- Missing core evidence lowers Evidence Quality but does not abort synthesis.
- Do not add Collection Attempt artifacts, Evidence Assessment schemas, provider-recipe
  abstractions, or new artifact versions.
- Keep the existing tolerant Source Plan v1/v2 reader. Add no new compatibility layer.
- Do not rewrite the historical run directories: their plans were produced after collection and
  cannot truthfully be converted into pre-collection intent.

Tests:

- Spy on ordering and prove plan creation precedes source-provider I/O.
- Prove plan contents are unchanged by differing collection outcomes.
- Prove assessment covers every planned lane.
- Prove missing core evidence degrades quality without aborting.
- Retain v1/v2 artifact-reader coverage.
- Prove new runs still write only the existing v2 artifact family.

### 3. Report integrity

Add these report fields:

```ts
type ReportIntegrity = "high" | "medium" | "low";

interface ResearchReport {
  readonly reportIntegrity?: ReportIntegrity;
  readonly researchQuality?: ReportIntegrity;
}
```

They are optional at tolerant read boundaries for historical reports and required for every new
report write. Research Quality is the worse of Evidence Quality and Report Integrity.

After initial schema-valid synthesis and before forecast disagreement:

1. Audit findings, scenarios, predictions, and summary.
2. If blocking violations exist, make one repair call with the synthesis model.
3. Revalidate the repaired report and rerun the deterministic audit.
4. Deterministically prune remaining blocking violations.
5. Validate and persist the final report.

Blocking violations:

- unknown or structurally ineligible source IDs
- numeric or technical findings, scenarios, or predictions without an eligible supporting source
- numeric or technical summary sentences, because summary has no citation field

Missing evidence-posture labels remain advisory telemetry and must not trigger pruning.
Deterministic checks establish structural eligibility only; do not claim semantic entailment.

Repair policy:

- At most one repair call; no repair retry.
- Use the synthesis model and existing provider timeout.
- Set `max_completion_tokens: 4000` where the provider supports it.
- Send only the report, violations, eligible source catalog, and repair instructions.
- Treat provider errors, timeout, malformed JSON, or schema-invalid output as repair failure.
- On failure, prune the original valid report and continue.

Pruning and grading:

- Remove an entire invalid finding, scenario, or prediction.
- Remove only offending summary sentences.
- `high`: final output is clean without deterministic pruning.
- `medium`: pruning occurred but required analytical sections remain.
- `low`: pruning leaves any required analytical section empty.
- Persist repair failure, repaired-item count, pruned-item count, and advisory-warning count in
  trace/analytics.
- Predictions removed here must never enter scoring.

Tests:

- Clean synthesis performs no repair.
- A valid repair produces high integrity.
- Repair timeout, error, malformed JSON, and schema-invalid output fall back to pruning.
- False-positive fixtures such as years and forecast-horizon wording are not pruned.
- Unsupported findings, scenarios, predictions, and summary sentences are removed.
- Advisory posture warnings do not trigger pruning.
- Medium and low grading follow the rules above.
- Research Quality is the worse of the two component grades.
- Pruned predictions are absent from persisted reports and scoring input.
- Historical reports without the new fields still load.

### 4. Scoring policy v3

#### Forecast contract

- Add `scoringPolicyVersion?: 3` to `Prediction`.
- Stamp version 3 during report assembly; never trust model-provided policy metadata.
- Keep `horizonTradingDays` and the existing forecast DSL. Under policy v3 it is the horizon count
  despite the legacy field name.
- Resolve a missing policy version permanently under policy v2.
- Never rewrite historical forecasts or already-resolved scores.
- Replace global interpretation through `SCORING_VERSION` with an explicit policy registry or
  equivalent resolver keyed by the Prediction's policy version.
- Persist the applied policy version in score results.

#### Policy v3 clocks

- Equity close forecasts: the Nth provider-observed session after the applicable anchor.
- Crypto close forecasts: the target UTC calendar date.
- Macro and IV forecasts: calendar days.
- Earnings forecasts: provider-observed equity sessions anchored to the declared earnings event.
- Exchange calendars may schedule retries but are not authoritative for outcomes.

#### Equity closes

- Request raw closes and split events for the complete origin-to-horizon window from one provider.
- Reconstruct split-adjusted, dividend-exclusive closes using the returned split events.
- Ignore dividends when constructing the price-only series.
- Do not mix providers within one resolution window.
- Treat a failed request, malformed split event, incomplete window, or inconsistent adjustment as
  unresolved rather than falling back to raw or total-return prices.

#### Retry state

- Persist `nextAttemptAt` for unresolved observation failures.
- Horizon-not-elapsed waits do not increment `attemptCount`.
- After the initial observation failure, schedule retries after 1, 3, and 7 days.
- Mark the forecast abandoned after the fourth failed observation fetch.
- Add `score --force` to bypass `nextAttemptAt` without changing scoring semantics.

#### Calibration

- Build the current dashboard from resolved policy-v3 forecasts only.
- Show resolved count, hit rate, Brier score, and explicit sample-size warnings.
- Do not generate a baseline-skill headline in new calibration summaries.
- Tolerate historical summaries that contain the legacy always-0.5 skill field.
- Do not implement empirical baseline skill until both conditions hold:
  - at least 100 resolved v3 forecasts overall
  - at least one event-kind × horizon stratum has 30 resolved forecasts
- Reopen the baseline design in a separate review when those thresholds are reached.

Tests:

- New Predictions are stamped v3; model-provided versions cannot override it.
- Missing-policy Predictions retain v2 behavior.
- Existing resolved score fixtures remain unchanged.
- Cover equity sessions, crypto UTC dates, macro/IV dates, and earnings anchors.
- Cover splits, ignored dividends, malformed metadata, incomplete windows, and provider mixing.
- Horizon waits do not consume attempts.
- Observation failures implement the 1/3/7-day schedule and fourth-failure abandonment.
- `score --force` bypasses timing only.
- New calibration excludes v2/unversioned results and omits baseline skill.
- Historical calibration summaries remain readable.

### 5. Deterministic peer comparability

- Fetch and normalize SEC SIC classification and provenance for the target and every candidate.
- A peer qualifies for the primary aggregate only when:
  - target and peer have matching two-digit SIC groups
  - peer market cap is inclusively 0.2x to 5x target market cap
  - peer annualized revenue is inclusively 0.2x to 5x target annualized revenue
  - existing freshness and valuation-input checks pass
- Missing SIC, market cap, annualized revenue, or freshness excludes the candidate with a
  deterministic reason.
- Require at least three qualifying peers before emitting median/IQR aggregates.
- Apply the same gates to mapped, registry-derived, cached, and model-proposed candidates.
- Retain rejected candidates and reasons as screening context.
- Business-model metadata may explain a candidate but cannot override a failed gate.

Tests:

- Same and different two-digit SIC groups.
- Inclusive 0.2x and 5x boundaries for market cap and revenue.
- Missing inputs produce explicit exclusions.
- Fewer than three qualifying peers remains screening-only.
- Three qualifying peers enables aggregate output.
- Cached and model-proposed candidates cannot bypass validation.

### 6. Run Chat search consent and capability

- Keep `MARKET_BOT_CONSOLE_CHAT_WEB_SEARCH` default enabled.
- Treat the setting as server permission; when enabled, live search is active by default only when
  the selected provider is Codex and the capability probe passes.
- Add cached provider capability metadata for live web search.
- For Codex, probe `codex --help` once per process for the advertised `--search` flag.
- Treat probe errors as `probe-failed`; ordinary chat remains usable without search.
- Do not rely only on a minimum Codex version.
- Use live search only when server policy and provider capability are both true.
- Keep non-Codex providers unsupported until explicitly implemented.
- Reject or ignore client escalation when server policy or capability denies search.
- Expose configured, supported, effective, and machine-readable reason states:
  - provider unsupported
  - capability probe failed
  - Codex search unsupported
  - disabled by server policy
  - active
- Add a visible Run Chat UI disclosure when live search is active. Disclose that search:
  - sends the question and selected run context to the configured external provider
  - may issue external web requests
  - may incur model or search cost
  - does not persist web results into Run Artifacts or Evidence Quality
- Update `docs/configuration.md`.

Tests:

- Server policy defaults enabled.
- Capability detection recognizes `--search`.
- Probe error and missing-flag states disable search without disabling chat.
- Codex invocation includes live-search flags only when both gates permit it.
- Unsupported providers report the correct reason.
- Client escalation is rejected server-side.
- Active-search disclosure is visible in the chat UI.
- Web findings remain ephemeral.

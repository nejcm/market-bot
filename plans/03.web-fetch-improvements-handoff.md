### Review amendments that override the original wording

- Treat this as provider-neutral web-gather resilience, not a simple Exa helper. Current code
  gates web-gather on `config.sourceOptions.exaApiKey` (`isWebGatherLoopEnabled`), so Firecrawl
  will not run when Exa is unconfigured unless that policy is deliberately changed. Decide and
  encode the policy explicitly:
  - fallback-only: Firecrawl runs only when Exa is configured but fails/thins out;
  - secondary-provider: Firecrawl may run when Exa is absent and `MARKET_BOT_FIRECRAWL_API_KEY`
    is present.
- Preserve source-unit accounting, but audit external provider attempts separately. Fallback
  does not spend another model tool unit, yet it can double paid provider requests and latency.
  Trace attempted provider, served provider, fallback reason (`hard-failure` / `empty` /
  `thin`), and Firecrawl credits when the API returns them.
- Verify Firecrawl request and response shapes against current docs before coding. In particular,
  do not assume `scrapeOptions.formats: ["markdown"]` is accepted for search; recent docs show
  object forms such as `{ "type": "markdown" }` in some contexts.
- Make SEC awareness measurable. Prompt guidance is allowed, but add a deterministic guard or
  audit classification for durable-profile background searches that are fully covered by the SEC
  packet. The intended savings should be observable in trace/analytics, not inferred from prompt
  text.
- Update docs by amending existing terms where present. `CONTEXT.md` already defines Web Gather,
  Web Evidence, Raw Web Snapshot, Model-Visible Web Text, and Web Subject Profile.
- ADR 0028 is the current canonical authority for web evidence and currently names Exa. Amend it
  or make the new ADR explicitly supersede only that provider-specific portion; do not leave two
  conflicting authorities.

### Context

On `equity <TICKER> --deep`, web evidence is a two-stage Exa pipeline
(`runWebEvidencePhase`). Investigation showed two of the three originally-proposed
improvements were already solved, and relocated the real opportunities:

- **"Let the 10-K answer many questions instead of one Exa query each"** is already done at
  the extraction layer. `runEvidenceRequestLoop` deterministically fetches the 10-K/10-Q and
  builds a bounded section packet *before* any Exa call
  (`src/sources/evidence-request-tools.ts:260`, `secFilingSectionPacket`), and Stage-2
  `web-subject-profile` answers all 11 company questions in **one** SEC-first model pass
  (`src/sources/extended-evidence/web-subject-profile.ts`). The remaining waste is that
  **Stage-1 web-gather is blind to what the filing already covers** and can burn its ≤4 Exa
  calls re-gathering durable facts. → **Fix ① (SEC-aware web-gather).**
- **"Batch questions in one fetch"** is already effectively done (Exa returns ≤8
  results/query; model emits multiple requests/round; extraction batches all questions). The
  only untapped batch (multi-URL `/contents`) saves latency but **zero** source-unit budget —
  the binding constraint — so it is **dropped**.
- **"Firecrawl as Exa fallback"** is a genuine resilience gain (Exa down / rate-limited /
  empty). Firecrawl v2 maps 1:1 onto the two Exa ops. → **Fix ③ (Firecrawl fallback).**

Intended outcome: fewer redundant Exa searches per deep equity run (①), and web evidence that
survives Exa outages / empty results by transparently falling back to Firecrawl (③). Both are
additive and behavior-preserving when their preconditions are absent.

Decisions locked with the user: scope = ① + ③ (skip ② batching); Firecrawl triggers on Exa
**hard failure OR empty/thin results** (after Exa's own widen retry).

### Fix ① — SEC-aware web-gather (reduce redundant Exa spend)

Make Stage-1 tell the model which durable business-profile areas the SEC filing packet
already sources, so Exa budget is spent only on recency / corroboration / genuine gaps.

**Files & changes**
- `src/research/research-context-types.ts:65` — extend `WebGatherContext` with an optional
  coverage field, e.g. `readonly secFilingCoverage?: { readonly present: boolean; readonly sections: readonly string[] }`.
- `src/research/web-gather-loop.ts` (`runWebGatherLoop` / `withWebGatherContext`, ~line 175) —
  derive coverage from `collectedSources.extendedSources`: reuse `isCompanyProfileSecSource`
  (`src/sources/extended-evidence/web-subject-profile.ts:147`) to detect present 10-K/10-Q
  packets, and parse the section labels already present by scanning each packet
  `source.snippet` for the `[Business] / [Risk Factors] / [MD&A] / [Segments] / [Notes]`
  markers emitted by `secFilingSectionPacket`. Pass the result into the context (company
  subjects only; empty/absent for crypto, themes, non-US, or when no packet was collected →
  identical to today's behavior).
- `prompts/web-gather/base.md` — add an instruction: when a SEC filing packet is present,
  treat the covered durable business-profile areas (business/segments/risk factors/MD&A) as
  **already sourced from the filing**; only issue `web_search` for recent developments,
  current context, corroboration, or areas the packet does **not** cover.
- Add deterministic audit/rejection logic for redundant durable-profile searches. At minimum,
  classify accepted/rejected `web_search` requests as `sec-covered-durable-profile` when the
  query/rationale targets only covered filing sections and lacks a recency, corroboration, or
  explicit-gap rationale. Prefer rejecting these requests with an audit reason; if rejection is
  too brittle, keep them accepted but emit trace telemetry so the Exa-reduction claim is
  measurable.

**Design notes**
- Detect coverage from the actual packet section markers rather than hard-coding a
  question-key map — less brittle, and honest about partial packets (e.g. filing with no
  parseable Segments section).
- No source-unit changes. Context + prompt is not enough by itself; add telemetry or a narrow
  validation guard so redundant-query reduction can be verified.

### Fix ③ — Firecrawl fallback for Exa (`web_search` and `web_fetch`)

Add Firecrawl v2 as a fallback provider behind the existing `executeWebGatherTool` seam,
triggered on Exa hard failure **or** empty/thin results. First make the web-gather policy
provider-neutral enough that the Exa enablement gate cannot silently suppress the fallback.

**Provider policy**
- Decide and encode whether Firecrawl is fallback-only or secondary-provider:
  - fallback-only keeps `MARKET_BOT_EXA_API_KEY` required for web-gather and uses Firecrawl only
    after configured Exa fails or returns empty/thin results;
  - secondary-provider changes `isWebGatherLoopEnabled` and unavailable gaps so web-gather can
    run with Firecrawl alone when Exa is absent.
- The originally approved intent was "Firecrawl fallback for Exa", so fallback-only is the
  conservative default unless the implementer deliberately chooses the stronger
  secondary-provider policy and updates tests/docs accordingly.
- Regardless of policy, do not let the current `isWebGatherLoopEnabled` Exa-key check make
  Firecrawl code unreachable in expected fallback cases.

**Config / plumbing**
- `src/config.ts` (~line 576, next to `exaApiKey`) — read `MARKET_BOT_FIRECRAWL_API_KEY` into
  `SourceOptions.firecrawlApiKey?`.
- `src/config.ts` `SourceOptions` interface + `src/sources/types.ts:36` `CollectContext` —
  add `firecrawlApiKey?: string`.
- `src/sources/collector.ts` (~line 1079, beside the `exaApiKey` spread in
  `createCollectContext`) — thread `firecrawlApiKey` into the context.

**Provider adapter**
- New `src/sources/firecrawl-web-tools.ts`: `firecrawlSearch` and `firecrawlScrape`.
  - Endpoints: `POST https://api.firecrawl.dev/v2/search` and `.../v2/scrape`; auth
    `Authorization: Bearer <key>`.
  - Verify current request shape from Firecrawl docs before implementation. The older expected
    search body was `{ query, limit, sources: ["web"], scrapeOptions: { formats: ["markdown"], onlyMainContent: true }, ...(tbs) }`;
    use the currently documented `formats` shape if it differs, and add tests around the exact
    request body emitted by the adapter.
    map freshness window → `tbs` (news/market → `qdr:m`, current-subject → `qdr:y`, background
    → omit). Response: `data.web[]` → `{ url, title, description→summary, markdown→text/snippet }`.
  - Verify scrape body shape too; expected body is `{ url, formats: ["markdown"], onlyMainContent: true }`; response
    `data.markdown`.
  - Call via `ctx.request.json({ url, adapter: "firecrawl-search" | "firecrawl-scrape", init, fetch })`
    so retry/caching/rawSnapshot all work. Mirror Exa's cache-key trick (`requestExaSearch` +
    `exaEndpointFetch`, `src/sources/web-gather-tools.ts:330,169`): encode query/url into the
    URL for the cache key, strip `url.search` before the real POST.
  - Normalize Firecrawl results into the existing `ExaResult`-shaped record so the same
    sanitize/emit path is reused.

**Wire the fallback into `src/sources/web-gather-tools.ts`**
- Generalize `exaSource` → `webSource(provider, ...)` and add a `provider` param to
  `outputFromResults` so fallback-sourced results are tagged `provider: "firecrawl"`
  (currently hard-codes `EXA_PROVIDER`, line 560). `rememberSurfacedUrls`,
  `sanitizeModelVisibleWebText`, and the freshness/sanitizer audit contracts are reused
  unchanged.
- `executeWebSearch` (line 228): after the existing Exa initial + widen path, if the final Exa
  outcome is a hard failure (`!isFetchJsonResult`) **or** usable
  `results.length < MIN_USABLE_SEARCH_RESULTS`, **and** `ctx.firecrawlApiKey !== undefined`,
  run `firecrawlSearch`; on success use its results (remember surfaced URLs, tag freshness with
  a `fallbackProvider: "firecrawl"` marker), else keep the Exa gap plus a Firecrawl gap.
- `executeWebFetch` (line 372): if the Exa `/contents` call fails or yields no usable content
  and the key is set, run `firecrawlScrape` on the (already-surfaced) URL.
- Budget/units unchanged — fallback runs inside the already-charged accepted request; the model
  contract (`web_search` / `web_fetch`, on-subject + surfaced-URL guards) is untouched. Add
  provider-attempt audit fields because external paid calls and latency are not unchanged.

**Gaps / audit**
- Emit provider-tagged gaps (`provider: "firecrawl"`, `capability: "web-gather"`) on Firecrawl
  failure, consistent with `webGatherGap`. Record which provider served each accepted request
  in the freshness audit for traceability.
- Fix `webGatherGap` before reuse: it currently accepts a `source` parameter but hard-codes
  `provider: EXA_PROVIDER`. Provider-tagged Firecrawl gaps need both `source` and `provider`
  to reflect Firecrawl, or a new provider-aware gap helper.
- Include `fallbackReason`, `attemptedProviders`, `servedProvider`, and `creditsUsed` when
  available. Treat Firecrawl `success: false` or missing expected data as provider gaps, not
  malformed Exa responses.

## Key facts established about the current pipeline (so the next agent needn't re-derive)

- Orchestration order (`src/research/orchestrator.ts:392,413`): `runEvidenceRequestLoop`
  (deterministic SEC 10-K/10-Q fetch) runs **before** `runWebEvidencePhase` (Exa web-gather +
  extraction). So SEC packets are always available to web-gather and extraction for US equities.
- SEC filing text packet is built by `secFilingSectionPacket`
  (`src/sources/evidence-request-tools.ts:260`) with per-section char budgets: Business 2000,
  Risk Factors 1500, MD&A 3000, Segments 1000, Notes 500. Sources get ids
  `extended-sec-edgar-<sym>-10k` / `-10q`; matched by `isCompanyProfileSecSource`.
- Note: `sec-edgar.ts` `collectSec` fetches only XBRL **companyfacts** (numbers), NOT filing
  text — the text packet comes from the evidence-request loop. Do not confuse the two.
- Web-gather budgets (`src/config.ts:161-163`): default maxRounds 2, maxToolCalls 4,
  sourceBudget 8; `web_search` = 2 units, `web_fetch` = 1 unit
  (`WEB_GATHER_TOOL_UNITS`). Budget is charged at validation time, before execution → Firecrawl
  fallback does not double-charge.
- Fetch abstraction: `ctx.request.json` / `.text` (`src/sources/types.ts`,
  `src/sources/collector.ts`) provide retry + caching + `rawSnapshot` + `SourceGap` on failure.
  Firecrawl must go through this to inherit those. `isFetchJsonResult` distinguishes success
  from `SourceGap`.
- Config → context plumbing pattern for provider keys: `exaApiKey` is read in `config.ts`
  `sourceOptions` (~line 576) and spread into `CollectContext` in `createCollectContext`
  (`collector.ts:~1079`). Mirror this for `firecrawlApiKey`.

## Firecrawl v2 API (must be re-verified before implementation)
- **Search:** `POST https://api.firecrawl.dev/v2/search`, `Authorization: Bearer <key>`.
  Body: `query` (≤500 chars), `limit`, `sources` (default `["web"]`; also `news`,`images`),
  `tbs` (time filter), `scrapeOptions.formats`, `onlyMainContent`.
  Response: `{ success, data: { web: [{ title, description, url, markdown, html, metadata }] }, creditsUsed }`.
  Warning: current docs may require object-shaped markdown formats in search scrape options
  rather than `["markdown"]`; adapter tests must pin the actual current shape.
- **Scrape:** `POST https://api.firecrawl.dev/v2/scrape`, same auth. Body: `url`, `formats`,
  `onlyMainContent` (default true), `maxAge`, `timeout`. Response:
  `{ success, data: { markdown, html, metadata } }`.

## Context

A prior session investigated discrepancies between AAPL Financial Lens stats across the last 3
runs (2026-06-27 ×2 and 2026-06-28). The June 27 runs had mixed-period SEC fundamentals: 2018
revenue (period end `2018-06-30`, 9 months) paired with 2026 income-statement items (6 months),
inflating margins and nearly doubling valuation multiples.

### Root cause (already fixed)

Commit `52b5daf` (*feat: enforce temporal integrity in research evidence*, 2026-06-28) rewrote
`src/sources/extended-evidence/sec-edgar.ts`:

- `latestFact` now sorts by **period end** descending (was **filed date** descending). The old
  sort picked a 2018 revenue fact because it had a recent `filed` date.
- Revenue is selected first; every other flow metric is filtered to the **same fiscal period**
  (`sameFiscalPeriod` + matching `end`/`start`) via a `flowPeriod` anchor.
- `isFactObservableAsOf` filters out facts filed or ending after `analysisAsOf`.

### Artifacts to read first (do not re-derive)

- `src/sources/extended-evidence/sec-edgar.ts` — the fixed selection logic (`selectMetric`,
  `isCurrentFlowFact`, `compareFactRecency`, `isFactObservableAsOf`, `summarizeSecFundamentals`)
- `tests/sources.test.ts` lines 1002–1218 — existing SEC temporal-integrity regression tests
- `src/sources/extended-evidence/valuation-comps.ts` lines 573–581 — `isFreshPeriodEnd` with
  180-day threshold (the pattern to reuse for Improvement B)
- `src/sources/cache.ts` lines 300–304 — stale-fallback payload wipe (cache already prevents
  stale payloads entering normalized evidence)
- `docs/adr/0028-deterministic-source-plan-subsystem.md` — temporal integrity ADR
- Run artifacts compared (for reference, not modification):
  - `data/runs/2026-06-28T10-42-57-445Z-a4e33cad/normalized/financial-lenses.json` (correct)
  - `data/runs/2026-06-27T18-09-16-064Z-094ffa60/normalized/financial-lenses.json` (corrupt)
  - `data/runs/2026-06-27T15-30-12-215Z-62ac0deb/normalized/financial-lenses.json` (corrupt, identical to above)

### Validation already performed

- Reproduced the AAPL corrupt payload and confirmed the current `summarizeSecFundamentals`

## Plan

Two improvements. Both touch `src/sources/extended-evidence/sec-edgar.ts` and
`tests/sources.test.ts`.

### Improvement A: Combined regression test (the exact AAPL bug shape)

**File:** `tests/sources.test.ts`, inside `describe("SEC fundamental evidence", ...)`

**Test name:** `"selects the correct revenue period and aligns flow metrics when an old fact has a newer filed date"`

**Payload** (mimics the corrupt AAPL company-facts; use the existing `secFact` helper at line 74):

- `Revenues` concept with two facts:
  - 2018 fact: `val: 202695000000`, `fy: 2018`, `fp: "Q3"`, `filed: "2026-05-01"`,
    `start: "2017-10-01"`, `end: "2018-06-30"` (old period, newer filed date)
  - 2026 fact: `val: 254940000000`, `fy: 2026`, `fp: "Q2"`, `filed: "2026-05-01"`,
    `start: "2025-10-01"`, `end: "2026-03-28"` (correct period)
- `GrossProfit` concept with one 2026 fact: `val: 124012000000`, same period as 2026 revenue
- `NetIncomeLoss` concept with one 2026 fact: `val: 71675000000`, same period

Pass `analysisAsOf: "2026-06-28T00:00:00.000Z"` as the second argument.

**Assertions:**

- `result?.metrics.revenue === 254940000000`
- `result?.metrics.revenuePeriodEnd === "2026-03-28"`
- `result?.metrics.grossProfit === 124012000000`
- `result?.metrics.grossProfitPeriodEnd === "2026-03-28"`
- `result?.metrics.netIncome === 71675000000`
- `result?.metrics.netIncomePeriodEnd === "2026-03-28"`
- `result?.gaps.some((g) => g.message.includes("Missing SEC company facts"))` is `false` for
  revenue, grossProfit, and netIncome (they are present; other missing metrics like
  `dilutedEps`, `cash` etc. will still produce a missing-facts gap — that's fine, assert only
  that revenue/grossProfit/netIncome are not in the missing list)

This test fails on pre-`52b5daf` code and passes on current code.

### Improvement B: Freshness guard on `revenuePeriodEnd`

**Files to modify:**

1. `src/config/shared/time.ts` — add `SEC_FRESHNESS_DAYS = 180`
2. `src/config/shared/index.ts` — re-export `SEC_FRESHNESS_DAYS`
3. `src/sources/extended-evidence/sec-edgar.ts` — add guard
4. `src/sources/extended-evidence/valuation-comps.ts` — replace local
   `const SEC_FRESHNESS_DAYS = 180` (line 28) with the shared import (DRY)
5. `tests/sources.test.ts` — add a test for the guard

#### Step 1: Shared constant

In `src/config/shared/time.ts`, add:

```ts
export const SEC_FRESHNESS_DAYS = 180;
```

In `src/config/shared/index.ts`, add to the re-exports:

```ts
export { SEC_FRESHNESS_DAYS } from "./time";
```

#### Step 2: Guard in `sec-edgar.ts`

Add import:

```ts
import { DAY_MS, SEC_FRESHNESS_DAYS } from "../../config/shared";
```

Add helper (near `isFactObservableAsOf`, ~line 315):

```ts
function isStalePeriodEnd(periodEnd: string, analysisAsOf: string): boolean {
  const periodMs = Date.parse(periodEnd);
  const cutoffMs = Date.parse(analysisAsOf);
  if (!Number.isFinite(periodMs) || !Number.isFinite(cutoffMs)) {
    return false;
  }
  return cutoffMs - periodMs > SEC_FRESHNESS_DAYS * DAY_MS;
}
```

In `summarizeSecFundamentals`, after the metric selection loop (after line 594, before the
`return` at line 596), add:

```ts
const staleRevenueGap =
  typeof metrics.revenuePeriodEnd === "string" &&
  analysisAsOf !== undefined &&
  isStalePeriodEnd(metrics.revenuePeriodEnd, analysisAsOf)
    ? [
        sourceGap({
          source: "sec-edgar",
          message: `Stale SEC revenue period: period end ${metrics.revenuePeriodEnd} exceeds ${SEC_FRESHNESS_DAYS} days`,
          provider: "sec-edgar",
          capability: "extended-evidence",
          cause: "provider-data-missing",
          evidenceQualityImpact: "extended-evidence-cap",
        }),
      ]
    : [];
```

Append `...staleRevenueGap` to the `gaps` array in the return object (line 602).

#### Step 3: DRY `valuation-comps.ts`

Replace line 28 `const SEC_FRESHNESS_DAYS = 180;` with:

```ts
import { DAY_MS, SEC_FRESHNESS_DAYS } from "../../config/shared";
```

(Already imports `DAY_MS`; add `SEC_FRESHNESS_DAYS`.)

Remove the local `const SEC_FRESHNESS_DAYS = 180;`.

#### Step 4: Test in `tests/sources.test.ts`

**Test name:** `"flags a stale SEC revenue period older than the freshness threshold"`

Payload: revenue fact with `end: "2025-01-01"`, `analysisAsOf: "2026-06-28T00:00:00.000Z"`
(age > 180 days).

Assertions:
- `result?.gaps.some((g) => g.message.includes("Stale SEC revenue period"))` is `true`
- `result?.metrics.revenue` is still populated (not suppressed)

Also add a **negative test:** `"does not flag revenue within the freshness threshold"`
- Revenue `end: "2026-03-28"`, `analysisAsOf: "2026-06-28T00:00:00.000Z"` (age < 180 days)
- Assert no gap message includes `"Stale SEC revenue period"`

#### Design decisions (locked)

- Stale revenue is **still emitted** in `metrics` — downstream consumers can use it, but the
  gap caps evidence quality. Matches `valuation-comps.ts` behavior.
- Guard only fires when `analysisAsOf` is provided (both production callers pass it).
- Future-dated periods (`ageMs < 0`) are not flagged — already filtered by `isFactObservableAsOf`.
- `cause: "provider-data-missing"` — best existing cause; no new union member needed.

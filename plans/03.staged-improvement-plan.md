# Staged Improvement Plan (supersedes plans/02)

## Purpose

Implement the remaining validated correctness and safety fixes from the audited plans/02
handoff, re-sequenced and scope-corrected. Stage numbering below is the new implementation
order; former plans/02 stage numbers are noted for traceability.

Status of former plans/02 stages: Stage 6 (Run Chat consent/capability) is implemented
(commits 112f2d4, fa91a3b). Stage 1 (external-text sanitization) is explicitly deferred
(see below). Stages 5, 2, 3, 4 are re-sequenced here as stages 1–4.

The repository is a single-user, US-centric research tool. Corpus at planning time:
6 run directories, 14 persisted forecasts, 0 resolved forecasts. Keep each change
independently reviewable and consistent with the scope-discipline rule.

## Repository constraints

- Research-only: no buy/sell/hold, sizing, or execution language.
- Predictions remain observable from public data.
- No secrets in code, tests, fixtures, traces, or errors.
- Bun and oxc only.
- No bundled refactors or speculative abstractions.
- Add tests with each behavioral change.
- Run `bun run check` at each stable completion point and before completion.
- Amend affected canonical ADRs in the same commit as the behavior change.

### 1. Deterministic peer comparability (was plans/02 stage 5)

Implementation notes: SIC is available from the already-fetched SEC submissions payload
(`src/sources/extended-evidence/sec-edgar.ts` fetches `data.sec.gov/submissions/CIK*.json`,
which carries `sic` and `sicDescription`) — capture it during normalization; add no new
endpoint. Market cap and annualized revenue are already computed per peer in
`src/sources/extended-evidence/valuation-comps.ts`.

- Normalize SEC SIC classification and provenance for the target and every candidate.
- A peer qualifies for the primary aggregate only when:
  - target and peer have matching two-digit SIC groups
  - peer market cap is inclusively 0.2x to 5x target market cap
  - peer annualized revenue is inclusively 0.2x to 5x target annualized revenue
  - existing freshness and valuation-input checks pass
- Missing SIC, market cap, annualized revenue, or freshness excludes the candidate with a
  deterministic reason.
- Require at least three qualifying peers before emitting median/IQR aggregates.
- Apply the same gates to mapped, registry-derived, cached, and model-proposed candidates.
- Retain rejected candidates and reasons as screening context.
- Business-model metadata may explain a candidate but cannot override a failed gate.

Tests:

- Same and different two-digit SIC groups.
- Inclusive 0.2x and 5x boundaries for market cap and revenue.
- Missing inputs produce explicit exclusions.
- Fewer than three qualifying peers remains screening-only.
- Three qualifying peers enables aggregate output.
- Cached and model-proposed candidates cannot bypass validation.

Docs: amend ADR 0006 (deterministic peer comparability); update the Peer Universe /
Valuation Evidence glossary entries in CONTEXT.md if gate semantics change their wording.

### 2. Frozen Source Plan before collection (was plans/02 stage 2)

This is an architecture change: ADR 0028 currently states source planning uses collected
sources; amend it in the same commit.

Keep the current v2 artifact family:

- `normalized/source-plan.json`
- `normalized/evidence-lanes.json`
- `normalized/source-ledger.json`

Split the current post-hoc builder (`src/research/source-plan.ts`, called from
`src/research/orchestrator.ts`) into two responsibilities:

- `buildSourcePlan(command, generatedAt)` creates the immutable v2 Source Plan before
  `collectSources`.
- `assessSourcePlan(plan, collectedSources)` creates Evidence Lanes and the Source Ledger
  after collection.

Requirements:

- Resolve the research command and checked-in research subject before creating the plan.
- Capture `generatedAt` and build the plan before the first source-provider I/O.
- Derive lane applicability and evidence class only from the resolved command, subject,
  asset class, depth, and checked-in policy.
- Never derive planned applicability or evidence class from collected outcomes,
  credentials, provider availability, or successful fetches.
- Use existing lane names as stable requirement identities.
- Grade coverage, gaps, freshness, and corroboration against the frozen plan.
- Missing core evidence lowers Evidence Quality but does not abort synthesis.
- Do not add Collection Attempt artifacts, Evidence Assessment schemas, provider-recipe
  abstractions, or new artifact versions.
- Keep the existing tolerant Source Plan v1/v2 reader. Add no new compatibility layer.
- Do not rewrite historical run directories: their plans were produced after collection and
  cannot truthfully be converted into pre-collection intent.

Tests:

- Spy on ordering and prove plan creation precedes source-provider I/O.
- Prove plan contents are unchanged by differing collection outcomes.
- Prove assessment covers every planned lane.
- Prove missing core evidence degrades quality without aborting.
- Retain v1/v2 artifact-reader coverage.
- Prove new runs still write only the existing v2 artifact family.

Docs: amend ADR 0028; update the Source Plan glossary entry in CONTEXT.md (it already says
"records intended coverage before synthesis" — tighten to "before collection").

### 3. Report integrity, deterministic-only (was plans/02 stage 3, narrowed)

Narrowed scope: no model repair call and no summary-sentence pruning in this stage (both
deferred; see Explicitly deferred). The "unknown source ID" blocking violation from the
original spec is dropped — report schema validation already rejects unknown source IDs.

Add these report fields:

```ts
type ReportIntegrity = "high" | "medium" | "low";

interface ResearchReport {
  readonly reportIntegrity?: ReportIntegrity;
  readonly researchQuality?: ReportIntegrity;
}
```

They are optional at tolerant read boundaries for historical reports and required for every
new report write. Research Quality is the worse of Evidence Quality and Report Integrity.

After initial schema-valid synthesis and before forecast disagreement, run a deterministic
Report Integrity Audit (distinct from the existing warn-only Post-Synthesis Audit):

1. Audit findings, scenarios, and predictions for blocking violations.
2. Deterministically prune blocking violations.
3. Validate and persist the final report.

Blocking violations (prune the entire invalid finding, scenario, or prediction):

- numeric or technical findings, scenarios, or predictions without an eligible supporting
  source (structural eligibility only; no semantic-entailment claims)

Advisory only (logged in analytics/trace, never pruned):

- numeric or technical summary sentences without citations (summary has no citation field)
- missing evidence-posture labels

Grading:

- `high`: final output is clean without pruning.
- `medium`: pruning occurred but required analytical sections remain.
- `low`: pruning leaves any required analytical section empty.
- Persist pruned-item count and advisory-warning count in trace/analytics.
- Predictions removed here must never enter scoring.

Tests:

- Clean synthesis produces high integrity with no pruning.
- False-positive fixtures (years, forecast-horizon wording) are not pruned.
- Unsupported numeric findings, scenarios, and predictions are removed.
- Summary sentences and posture warnings are advisory only and never pruned.
- Medium and low grading follow the rules above.
- Research Quality is the worse of the two component grades.
- Pruned predictions are absent from persisted reports and scoring input.
- Historical reports without the new fields still load.

Docs: amend ADR 0011; add Report Integrity Audit, Report Integrity, and Research Quality
glossary entries to CONTEXT.md; keep the Post-Synthesis Audit entry warn-only and
cross-reference the new audit.

### 4. Scoring policy v3 (was plans/02 stage 4, split into four commits)

Land 4a–4d as separate reviewed commits, in order.

#### 4a. Policy registry, stamping, and v3 clocks

- Add `scoringPolicyVersion?: 3` to `Prediction`.
- Stamp version 3 during report assembly; never trust model-provided policy metadata.
- Keep `horizonTradingDays` and the existing forecast DSL. Under policy v3 it is the
  horizon count despite the legacy field name.
- Resolve a missing policy version permanently under policy v2.
- Never rewrite historical forecasts or already-resolved scores.
- Replace global interpretation through `SCORING_VERSION` with an explicit policy registry
  or equivalent resolver keyed by the Prediction's policy version.
- Persist the applied policy version in score results.

Policy v3 clocks:

- Equity close forecasts: the Nth provider-observed session after the applicable anchor.
- Crypto close forecasts: the target UTC calendar date.
- Macro and IV forecasts: calendar days.
- Earnings forecasts: provider-observed equity sessions anchored to the declared earnings
  event.
- Exchange calendars may schedule retries but are not authoritative for outcomes.

Tests: new Predictions are stamped v3 and model-provided versions cannot override;
missing-policy Predictions retain v2 behavior; existing resolved score fixtures unchanged;
cover equity sessions, crypto UTC dates, macro/IV dates, and earnings anchors.

#### 4b. Retry state

- Persist `nextAttemptAt` for unresolved observation failures.
- Horizon-not-elapsed waits do not increment `attemptCount`.
- After the initial observation failure, schedule retries after 1, 3, and 7 days.
- Mark the forecast abandoned after the fourth failed observation fetch.
- Add `score --force` to bypass `nextAttemptAt` without changing scoring semantics.

Tests: horizon waits do not consume attempts; 1/3/7-day schedule and fourth-failure
abandonment; `score --force` bypasses timing only.

#### 4c. Equity split-adjusted closes

- Request raw closes and split events for the complete origin-to-horizon window from one
  provider.
- Reconstruct split-adjusted, dividend-exclusive closes using the returned split events.
- Ignore dividends when constructing the price-only series.
- Do not mix providers within one resolution window.
- Treat a failed request, malformed split event, incomplete window, or inconsistent
  adjustment as unresolved rather than falling back to raw or total-return prices.

Tests: splits, ignored dividends, malformed metadata, incomplete windows, provider mixing.

#### 4d. Calibration v3 presentation

- Build the current dashboard from resolved policy-v3 forecasts only.
- Show resolved count, hit rate, Brier score, and explicit sample-size warnings.
- Do not generate a baseline-skill headline in new calibration summaries.
- Tolerate historical summaries that contain the legacy always-0.5 skill field.
- Do not implement empirical baseline skill until both hold:
  - at least 100 resolved v3 forecasts overall
  - at least one event-kind × horizon stratum has 30 resolved forecasts
- Reopen the baseline design in a separate review when those thresholds are reached.

Tests: new calibration excludes v2/unversioned results and omits baseline skill; historical
calibration summaries remain readable.

Docs (across 4a–4d, amended with the sub-stage that changes each behavior): amend ADR 0004
(policy v3, clocks, adjustment, retry, calibration presentation); amend the Calibration
glossary entry in CONTEXT.md (drop "Brier skill score versus a 0.5 baseline" from the
current-dashboard description); add a Scoring Policy glossary entry.

## Explicitly deferred

- **Universal external-text sanitization (former plans/02 stage 1).** Full spec preserved
  below. Revisit trigger: before any run automation/scheduling lands, or on the first
  observed prompt-injection artifact in a run.
  - Reuse or generalize the existing model-visible web-text sanitizer only as required for
    reuse; no terminology-only rename.
  - Sanitize all provider-normalized prose before it enters model context: titles,
    publishers, summaries/descriptions, snippets, SEC filing excerpts.
  - Cover MarketAux, Finnhub, Yahoo, Massive, SEC, and Exa normalization paths.
  - Validate rather than prose-sanitize identifiers, dates, and URLs.
  - Preserve raw provider snapshots byte-for-byte.
  - Preserve safe retained text and existing sanitizer telemetry conventions.
  - When no safe prose remains, omit it from model-visible evidence and emit a Source Gap
    without discarding the raw snapshot.
  - Tests: provider fixture per named path; instruction spans and unsafe HTML never reach
    model-visible values; safe remainder and sanitizer counts; empty sanitized content
    creates a gap; raw snapshots unchanged.
- **Report repair call and summary-sentence pruning (from former stage 3).** Revisit
  trigger: real runs show the stage-3 deterministic audit pruning (or flagging summaries)
  often enough that a one-shot synthesis-model repair pass would preserve meaningful
  content. Preserved spec: at most one repair call, synthesis model, existing provider
  timeout, `max_completion_tokens: 4000` where supported; send only report, violations,
  eligible source catalog, and repair instructions; provider error / timeout / malformed
  JSON / schema-invalid output is repair failure; on failure prune the original valid
  report and continue.
- `ResearchRunLifecycle.execute()` extraction: revisit only on a behavioral defect or
  repeated orchestration change.
- Collection Attempt and Evidence Assessment artifact redesign.
- Empirical walk-forward baseline skill until the stage-4d sample thresholds and a new
  design review.
- Synthetic 10,000-run benchmarks, incremental scoring, and index redesign: profile the
  real corpus only after repeated user-visible latency.

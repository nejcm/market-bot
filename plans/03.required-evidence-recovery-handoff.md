## 2. Approved plan

# Required Evidence Recovery and Subject Evidence Profile v4

## Summary

Make `equity --deep` evidence coverage deterministic. Required
filings/reports and eleven company-profile questions are defined by Source Plan
v3, recovered through official documents, Exa, or structured provider search,
and never marked satisfied without fetched evidence and verified anchors.

Runs continue with explicit gaps when recovery fails. Recovery is enabled by
default only after all phases pass review.

## Interfaces and artifacts

- Add Source Plan v3 `EvidenceRequirement`:
  - States: `satisfied`, `unresolved`, `not-applicable`.
  - Annual report: core.
  - Applicable interim report and eleven profile questions: material.
  - Include source IDs, authority/detail level, recovery-attempt IDs, and
    unresolved reason.
- Replace Web Subject Profile with Subject Evidence Profile v4:
  - Answers use `supported` or `unresolved`.
  - Supported answers require source IDs and verified source-text anchors.
  - Record `quantified` versus `qualitative` detail.
  - Persist only cited filing/report metadata in `basisDocuments`.
- Write:
  - `normalized/subject-evidence-profile.json`
  - `normalized/evidence-recovery.json`
  - Source Plan v3 artifacts
  - Hashed raw HTML/PDF documents under the run’s raw directory
- Read legacy `web-subject-profile` v3 artifacts for display only; never reuse
  or reconcile them.
- Add separate `WebSearchProvider` capability returning structured URLs,
  titles, timestamps/page age, query audit, and provider metadata.
- Add minimal configuration:
  - recovery disable flag
  - document-search limit, default `2`
  - profile-search limit, default `5`
  - profile-page-fetch limit, default `10`
- Keep four official-document downloads, 25 MB/400-page extraction, and
  80-page OCR limits as versioned policy.

### 2. Deterministic official-document recovery

- Remove mandatory filing retrieval from model discretion.
- For SEC filers:
  - Fetch fresh submissions metadata.
  - Require latest 10-K.
  - Require latest 10-Q only when metadata lists one after the latest 10-K.
  - Fetch annual/interim documents concurrently.
  - Fall back to an issuer-hosted copy only when form, dates, period, and
    accession identity match.
- For international listings:
  - Resolve and validate the issuer’s official domain.
  - Discover latest official annual and applicable interim reports.
  - Validate issuer identity, document type, reporting period, and publication
    date.
  - Persist validated issuer-domain mappings with provenance, 30-day TTL, and
    revalidation.
- Use pinned `unpdf` and `tesseract.js`.
  - Extract embedded text first.
  - OCR only when needed.
  - Support the approved recognized-market language set.
  - Preserve original-language anchors; profile answers remain English.
- Build bounded section packets covering business description,
  segments/geography, MD&A, risks, and relevant notes.
- Sanitize all model-visible text while retaining original raw bytes.
- Apply one-hour freshness to SEC metadata and selected pages.

Review, run document/PDF/OCR tests, run `bun run check`, fix findings, commit.

### 4. Two-pass extraction and reconciliation

Pipeline:

1. Retrieve official documents.
2. Run optional web gather.
3. Produce preliminary profile coverage.
4. Recover unresolved requirements.
5. Produce final profile.
6. Reconcile supported answers.
7. Render and persist artifacts.

Rules:

- Each extraction pass gets one schema-repair retry using unchanged evidence.
- If final extraction fails, retain the validated preliminary profile.
- A supported answer requires citations plus verified source-text anchors.
- “Not disclosed” remains unresolved and cannot clear a framework gap.
- Primary evidence needs one source; secondary evidence needs two independent
  sources.
- Filing-backed answers remain valid until newer filing metadata appears.
- Primary-web answers: 30 days.
- Secondary-only answers: 7 days.
- Recent-event evidence: 24 hours.
- `Basis` lists only annual/interim documents actually cited by supported
  answers.
- Missing annual evidence applies a core Evidence Quality cap.
- Missing applicable interim or profile-question evidence applies a material
  cap.
- `not-applicable` has no quality impact.

Review reconciliation/report tests, run `bun run check`, fix findings, commit.

## 3. Complete grilling Q&A

### Q1 — Exact filing fallback

When SEC retrieval fails, should fallback require the exact filing document?

**Answer:** Yes. Use this hierarchy:

1. SEC submissions/archive.
2. Issuer investor-relations filing.
3. Exa/web search to locate the exact filing.
4. Web evidence may fill individual profile questions.
5. Summaries/articles never replace a 10-K.

Canonical term: **Required Evidence Recovery**.

### Q2 — Definition of necessary data

What defines “all necessary data” for a deep equity run?

**Answer:** A fixed, code-defined contract. Recovery targets every unmet
requirement before analysis; the LLM does not decide whether the requirements
exist.

### Q3 — Failure after recovery

What happens after every approved fallback is exhausted?

**Answer:** Continue degraded. Preserve explicit unresolved gaps and prevent
unsupported profile answers or reconciliation.

### Q4 — Recovery budgets

Should mandatory recovery share optional evidence-request/web-gather budgets?

**Answer:** No. It receives a reserved, deterministic, bounded budget so
optional requests cannot starve required evidence.

### Q5 — Initial contract scope

Which evidence does the first Required Evidence Contract cover?

**Answer:** Filings/reports plus every company-profile question. Existing market,
news, options, earnings, and valuation systems retain ownership of their lanes.

### Q6 — Qualitative recovery strategy

Who formulates web recovery searches?

**Answer:** Hybrid recovery. Run deterministic issuer/question queries first,
then permit one bounded LLM-proposed query for remaining gaps. Code validates
scope and a search adapter executes the request.

### Q7 — Web proof threshold

Can Exa/provider search snippets resolve requirements?

**Answer:** No. Search results only discover URLs. The selected page must be
fetched, sanitized, cited, and pass authority rules.

### Q8 — Source authority

Which fetched web sources can satisfy company-profile requirements?

**Answer:** Tiered authority:

- Primary sources can satisfy applicable facts.
- Reputable secondary sources may satisfy qualitative questions.
- Secondary sources cannot replace filings or independently establish
  filing-derived financial percentages.

### Q9 — Alternate filing copy

Which document can count as the exact 10-K/10-Q when SEC retrieval fails?

**Answer:** SEC-hosted or issuer-hosted copies only, with matching form, filing
date, reporting period, and accession identity. Third-party mirrors remain
discovery aids.

### Q10 — Profile question state

How is an unanswered question represented?

**Answer:** Explicit status in a versioned artifact: `supported` or
`unresolved`. Only supported answers carry accepted claims and can clear
Business Framework gaps.

### Q11 — Secondary corroboration

What corroboration is required for secondary-only qualitative evidence?

**Answer:** Two independent fetched sources. One qualifying primary source is
sufficient.

### Q12 — Source independence

How does code determine independence?

**Answer:** Distinct publisher ownership/registrable domains, excluding
canonical duplicates and detected syndicated copies. The model does not decide.

### Q13 — Pipeline shape

How does the system determine what needs web recovery?

**Answer:** Two-pass profile:

1. Preliminary coverage extraction.
2. Targeted recovery.
3. Final extraction over the completed evidence set.

### Q14 — Recovery cost priority

What do reserved defaults optimize for?

**Answer:** Coverage first. Every unresolved question group receives a genuine
attempt.

### Q15 — Profile search/fetch limits

What are the defaults?

**Answer:** Five profile searches and ten full-page fetches. Four searches cover
deterministic question groups; one is reserved for an LLM-proposed follow-up.

### Q16 — Search-provider abstraction

What happens when Exa is not configured?

**Answer:** Introduce a provider-neutral search capability. Exa is preferred,
and structured search from the selected model provider is allowed.

### Q17 — Search adapters

Which search sources must be supported?

**Answer:** Exa and the currently selected search-capable model provider.
OpenAI and Anthropic receive explicit structured adapters. Codex/arbitrary
compatible providers qualify only if they return structured source metadata.

### Q18 — Freshness definition

What does “fresh and latest” mean?

**Answer:** Requirement-specific freshness:

- Latest authoritative filing/report available at `analysisAsOf`.
- Live search.
- Selected pages refetched when stale.
- Publication/update/fetch timestamps remain distinct.
- Older durable disclosures remain usable when they are still the latest
  authoritative evidence.

### Q19 — Provider order

How are Exa and native provider search combined?

**Answer:** Exa first. Native provider search is used for unresolved gaps,
corroboration, or Exa failure. Do not duplicate every query.

### Q20 — Provider capability threshold

What must a native provider return?

**Answer:** Structured URLs, titles, query/tool audit metadata, and timestamps
or page age when available. Generated prose and parsed links do not qualify.

### Q21 — Unsupported provider behavior

What if Exa is unavailable and the selected provider lacks structured search?

**Answer:** Continue with direct official retrieval, then preserve explicit
provider-unavailable and unresolved gaps. Do not fail the run or parse generated
links.

### Q22 — Network cache age

What freshness limits apply to mandatory recovery inputs?

**Answer:** One hour for SEC metadata and selected pages. Recovery search itself
is live.

### Q23 — Profile reuse

How should profile reuse work?

**Answer:** Per-question freshness. Revalidate current-sensitive questions
without discarding still-valid durable answers. Newer filing metadata
immediately invalidates affected filing-backed answers.

### Q24 — Answer TTL basis

What determines answer reuse lifetime?

**Answer:** Source-based:

- Filing-backed: until newer periodic filing/report metadata.
- Primary web: 30 days.
- Secondary-only: 7 days.
- Recent-event evidence: 24 hours.

### Q25 — Quarterly requirement

When is a 10-Q required?

**Answer:** Only when authoritative SEC metadata lists a 10-Q after the latest
10-K. Otherwise quarterly coverage is `not-applicable`, not missing.

### Q26 — Listing scope

Does the contract cover international equities?

**Answer:** Yes, use global equivalents.

### Q27 — Global filing authority

What happens without a dedicated regulator adapter?

**Answer:** Prefer a regulator adapter, then accept a fetched issuer-hosted
annual/interim report after validating issuer, period, publication date, and
document type.

### Q28 — Initial regulator adapters

Which regulator adapters ship initially?

**Answer:** SEC plus global validated issuer fallback. Keep an extensible
regulator registry for later additions.

### Q29 — Filing text projection

How much document text enters profile extraction?

**Answer:** A deterministic bounded section packet covering Business, MD&A,
Risk Factors, segment/geography disclosures, and relevant notes. Do not send a
single 6,000-character excerpt or the complete document.

### Q30 — PDF handling

How are issuer-report PDFs processed?

**Answer:** Extract embedded text first and use bounded OCR only for image-based
pages. Preserve OCR provenance and quality disclosures.

### Q31 — PDF/OCR runtime

Where does extraction run?

**Answer:** Locally under Bun with pinned JavaScript dependencies. Use `unpdf`
and `tesseract.js`; do not upload reports to model providers.

### Q32 — Document limits

What safety limits apply?

**Answer:** 25 MB, 400 pages for embedded-text extraction, and 80 prioritized OCR
pages. Cap hits produce explicit gaps.

### Q33 — Issuer-domain validation

How is an official issuer domain established?

**Answer:** Providers may propose domains. Code grants primary status only after
regulator metadata, canonical issuer links, company identity, and redirects
validate.

### Q34 — Issuer-domain cache

Should validated domains persist?

**Answer:** Yes. Store provenance in a learned cache with a 30-day TTL and
revalidate identity/redirects on read.

### Q35 — Architectural owner

Where does Required Evidence live?

**Answer:** Source Plan v3. A Source Plan lane remains a provider-neutral
channel; an Evidence Requirement is a concrete document/fact inside that lane.

### Q36 — Requirement states

Which terminal states are persisted?

**Answer:** `satisfied`, `unresolved`, and `not-applicable`. Attempt history
contains lifecycle detail.

### Q37 — Profile migration

How are existing v3 artifacts handled?

**Answer:** Read-only legacy. Write v4, continue rendering v3, but never reuse v3
or let it perform new reconciliation.

### Q38 — Canonical profile name

What replaces Web Subject Profile?

**Answer:** **Subject Evidence Profile** across subject kinds. Web Subject
Profile becomes a legacy reader term.

### Q39 — Persisted key migration

How do file paths/report extras migrate?

**Answer:** New write, legacy read:

- `normalized/subject-evidence-profile.json`
- `extras.subjectEvidenceProfile`

Do not dual-write legacy keys.

### Q40 — Basis semantics

What does the rendered Basis line represent?

**Answer:** Only filing/report documents actually cited by supported answers,
including document type, period, publication/filing date, and authority tier.

### Q41 — Report disclosure

How much recovery detail appears in the human report?

**Answer:** Compact coverage summary, unresolved requirement names, filing
status, and cited Basis. Full attempts stay in normalized artifacts and trace.

### Q42 — Remaining framework gaps

Should `whatItDoes` and `geography` receive deterministic framework gaps?

**Answer:** Yes. Add `business-description` and `geographic-mix`; only supported
answers clear them.

### Q43 — Mandatory profile components

Are subject summary, fact ledger, and recent events mandatory requirements?

**Answer:** No. Requirements are official documents plus eleven company
questions. Summary/fact ledger are derived; recent events remain owned by
existing news/current-context lanes.

### Q44 — Segment/geography detail

Can qualitative evidence satisfy segment/geography requirements?

**Answer:** Yes, after authoritative sources do not disclose percentages. Record
the lower `qualitative` detail level; prefer `quantified`.

### Q45 — Supported-answer proof

What proof must a supported answer carry?

**Answer:** Source IDs plus short source-text anchors verified against normalized
fetched text. Secondary support requires anchors from both independent sources.

### Q46 — Raw document storage

How are fetched documents retained?

**Answer:** Hashed raw files under the run raw directory, with SHA-256, content
type, byte count, URL, and retrieval metadata. Normalized artifacts reference
them.

### Q47 — Document sanitization

Are official documents exempt from prompt-injection sanitization?

**Answer:** No. Sanitize all model-visible text while retaining unsanitized raw
bytes. Authority affects evidentiary weight, not instruction handling.

### Q48 — Provider interface

How does native search integrate?

**Answer:** A separate `WebSearchProvider` interface. Exa, OpenAI, and Anthropic
adapters normalize results independently of ordinary model generation.

### Q49 — Existing web-gather loop

Does recovery replace optional web gather?

**Answer:** No. Recovery owns mandatory requirements. Optional gather remains
for broader context/recent developments and must pass recovery validation before
its sources can satisfy requirements.

### Q50 — Pipeline ordering

When does preliminary coverage run?

**Answer:** Official documents, then optional web gather, then preliminary
coverage, targeted recovery, and final extraction.

### Q51 — Final extraction failure

What survives malformed final output?

**Answer:** Keep the validated preliminary profile, record failure, and do not
attach newly recovered evidence to unsupported answers.

### Q52 — Schema repair

How many extraction repair retries are allowed?

**Answer:** One per extraction pass, using unchanged evidence and exact
validation errors. No extra search.

### Q53 — Evidence Quality

How do unresolved requirements affect quality?

**Answer:**

- Missing annual filing/report: core cap.
- Missing applicable interim/report or profile question: material cap.
- `not-applicable`: no impact.

### Q54 — Configuration surface

How configurable is recovery?

**Answer:** Minimal environment surface: enable/disable plus search/fetch caps.
Authority, freshness, document/OCR, quality, and retry rules remain versioned
code policy.

### Q55 — Run scope

Which runs execute recovery?

**Answer:** `equity --deep` instrument runs only. Brief, market overview,
alpha-search, crypto, and thematic runs remain unchanged.

### Q56 — Global search budget

Does global issuer-report discovery share the five profile searches?

**Answer:** No. Reserve two document-discovery searches plus five profile
searches and ten profile-page fetches.

### Q57 — Document download budget

How many official-document candidates may be downloaded?

**Answer:** Four, allowing an alternate annual and interim candidate.

### Q58 — Native search model

Which configured model is used?

**Answer:** Quick model first, synthesis model only if quick lacks structured
search support. No separate model setting.

### Q59 — Unknown publication date

Can an undated page satisfy a requirement?

**Answer:** Only a validated primary issuer/regulator page may support a durable
fact using fetch time. Undated secondary and recent-event pages cannot satisfy
requirements.

### Q60 — Secondary reputation

How is “reputable secondary” determined?

**Answer:** A versioned, checked-in publisher/ownership registry. Unknown domains
may provide context but cannot satisfy requirements.

### Q61 — Initial publisher registry breadth

How broad is the initial registry?

**Answer:** Conservative global set, grouped by ownership, expanded only through
reviewed code changes.

### Q62 — Rollout default

Is recovery enabled by default?

**Answer:** Yes for deep-equity runs, but activate only after every implementation
phase passes. Disabling recovery still exposes unresolved requirements and
quality effects.

### Q63 — Quantification upgrades

Can spare budget improve already qualitative segment/geography answers?

**Answer:** Yes, after all unresolved requirements receive attempts. Missing
coverage always has priority.

### Q64 — Delivery strategy

How is the work delivered?

**Answer:** Phased commits, review and `bun run check` after every phase, default
activation only at the end.

### Q65 — Live acceptance symbols

Which symbols define manual acceptance?

**Answer:** AAPL, RR.L, and RY.TO.

### Q66 — OCR language policy

How are non-English scanned reports handled?

**Answer:** Language-aware OCR using pinned, cached, allowlisted language packs.
Unsupported languages produce explicit gaps.

### Q67 — Initial OCR language set

Which languages are approved?

**Answer:** English, French, German, Spanish, Italian, Portuguese, Dutch, Polish,
Swedish, Norwegian, Danish, Finnish, Japanese, Korean, Simplified Chinese,
Traditional Chinese, Hebrew, and Turkish.

### Q68 — Multilingual evidence

How does non-English evidence support an English profile?

**Answer:** Preserve the verified source-language anchor and language tag. The
model may author an English answer, but its translation is not treated as source
text.

### Q69 — Final approval

Should the decisions be locked into an implementation-ready plan?

**Answer:** Yes. The plan in section 2 is approved.

- Research-only boundary remains non-negotiable.

- Add tests with every behavioral change.
- Do not bypass hooks.

  OpenAI provider uses Chat Completions and does not expose structured search

  including provider errors returned inside successful HTTP responses.
- Current Codex JSONL exposes web-search query/action items but not a guaranteed
  structured result/source contract; do not treat it as an evidence search
  adapter unless that contract is added and tested.
- Prefer a dedicated recovery coordinator/module over adding more branches to
  the already-large orchestrator.
- Move filing basis ownership into the Subject Evidence Profile rather than
  scanning all Extended Evidence during rendering.
- Keep legacy readers isolated; do not spread v3/v4 branching through business
  logic.
